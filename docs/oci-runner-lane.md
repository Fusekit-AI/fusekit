# OCI Runner Lane

FuseKit's OCI lane is the clean-room execution path for users who want the app setup worker to run on a disposable cloud VM instead of their local computer. It is not a way to bypass Oracle signup, MFA, payment-card verification, capacity limits, provider consent, or billing controls. Those remain human gates. After the gate is passed, FuseKit should do the rest.

## Product Shape

Default runner behavior should mirror the LLM lane:

```text
fusekit launch /path/to/app --runner auto
```

`auto` should resolve in this order:

1. Use an already configured explicit runner from the app or user's FuseKit profile.
2. Use an existing OCI account if FuseKit can find an approved OCI auth profile.
3. Guide the user through OCI Free Tier signup/login and API authorization.
4. Fall back to local only when the user chooses local or OCI cannot be provisioned.

Explicit lanes:

```text
fusekit launch /path/to/app --runner local
fusekit launch /path/to/app --runner oci-free
fusekit launch /path/to/app --runner oci-existing
```

V1 implements `local`, `oci-cloud-shell`, `oci-free`, and `oci-existing`. BYOC is reserved as a provider plugin lane and is not exposed as a selectable runner without a real SSH/bootstrap/detonation adapter.

## OCI Auth Modes

OCI auth needs the same human-gated fallback style as LLM auth:

```text
--oci-auth-mode auto
--oci-auth-mode existing-config
--oci-auth-mode browser-session
--oci-auth-mode api-key-upload
```

`auto` should try:

1. Existing config from `OCI_CONFIG_FILE`, `~/.oci/config`, or a vault-stored OCI profile.
2. OCI CLI security-token browser auth when available. This is the default no-paste path: FuseKit launches `oci session authenticate`, Oracle handles login/MFA/account gates, and FuseKit captures the generated local security-token profile into the encrypted vault.
3. API signing-key setup: FuseKit generates a private/public RSA signing key, opens the OCI API Keys console page, drives the public-key upload when browser automation can identify the control, captures the approved config internally, then encrypts the private key and config in the vault.
4. Guided signup/login: FuseKit opens Oracle Cloud signup, waits at contact/card/MFA/region gates, then resumes API-key setup.

Never scrape card data, password-manager data, MFA codes, or CAPTCHA results. Never create multiple free accounts. Oracle's Free Tier FAQ says one Free Trial or Always Free account is permitted per person, contact/billing data must be valid and consented, and credit/debit card information is used for identity verification.

Useful official references:

- OCI Free Tier FAQ: https://www.oracle.com/cloud/free/faq/
- OCI API signing keys: https://docs.oracle.com/iaas/Content/API/Concepts/apisigningkey.htm
- OCI CLI token auth: https://docs.oracle.com/en-us/iaas/Content/API/SDKDocs/clitoken.htm

## Provisioning Plan

After authorization, FuseKit creates a tagged, disposable OCI workspace. There is no FuseKit-originated provisioning approval gate; the user's OCI account/login/API-key flow is the service-side authorization boundary. FuseKit may still stop at service-originated gates such as OCI signup, MFA, payment-card verification, capacity exhaustion, identity checks, and API authorization.

1. Compartment: the current tenancy/root compartment by default, matching the OCI
   console launch path. A child compartment `fusekit-<timestamp>-<short-id>` is
   available only through explicit isolated mode.
2. VCN with internet connectivity.
3. Public subnet.
4. Internet gateway and route rule.
5. Network security group:
   - SSH from the detected client IP only when possible.
   - HTTP/HTTPS only if the setup plan requires live preview callbacks.
   - Egress for provider setup and package installs.
6. Ephemeral SSH key pair, generated by FuseKit.
7. Compute shape:
   - Default to the x86_64 `VM.Standard.E5.Flex` shape with 2 OCPU and 24 GB RAM so OpenClaw, browser tooling, package installs, and runner gates have enough memory while matching the console-offered E5 baseline.
   - Fall back only across x86_64 Flex shapes with the same 2 OCPU and 24 GB sizing, currently `VM.Standard.E4.Flex` and `VM.Standard3.Flex`.
   - Reject known Arm/Ampere A1 shapes instead of silently falling back to them. FuseKit's remote bootstrap path assumes an x86_64 runner.
   - Allow explicit non-free/paid x86_64 shapes when selected by the user through OCI account authorization and CLI options; FuseKit should prefer a working live runner over adding its own approval gate. The default runner shape may consume paid capacity depending on the OCI tenancy.
8. Canonical Ubuntu 24.04 LTS image with cloud-init by default for the runner bootstrap because it is in Playwright's supported Linux matrix and keeps browser dependencies on the `apt` path. FuseKit falls back to Ubuntu 22.04 LTS, then an unpinned Ubuntu image, then Oracle Linux only when Ubuntu is unavailable for the selected x86 shape.
9. VM launch options aligned to the current OCI instance flow:
   - IMDSv1 disabled so instance metadata requires the IMDSv2 authorization header.
   - OCPU baseline left unset so OCI applies the selected E5 Flex shape default.
   - In-transit encryption enabled for paravirtualized boot-volume attachment.
   - Restore instance lifecycle state after infrastructure maintenance.
10. Boot volume marked delete-on-termination.

FuseKit leaves OCI placement defaults such as fault domain and migration behavior to Oracle.
It follows the console's private-VNIC launch path first, then attaches the runner NSG and
assigns an ephemeral public IPv4 address after the instance is running. This keeps the
compute capacity request close to the OCI console defaults while still making SSH,
control-room, and noVNC reachable for the live run. Before launch, FuseKit asks OCI for a
compute capacity report using the root tenancy compartment as required by the API, then
attempts the launch only on x86 runner options.
Using the root/current compartment by default avoids child-compartment IAM, quota, and
propagation mismatches where OCI reports capacity in the tenancy but rejects launch in a
freshly-created compartment.

Capacity failures are a service-side gate with an automatic retry loop, not a terminal failure. OCI can return out-of-capacity responses for individual shapes and availability domains. FuseKit should retry configured x86_64 runner shapes at the recommended runner size, then use the selected paid/BYOC x86_64 shape when the user has configured one.

Useful official references:

- OCI Always Free resources: https://docs.oracle.com/en-us/iaas/Content/FreeTier/freetier_topic-Always_Free_Resources.htm
- OCI creating instances: https://docs.public.content.oci.oraclecloud.com/en-us/iaas/Content/Compute/Tasks/launchinginstance.htm

## Remote Bootstrap

Cloud-init should create a self-contained runner with no dependency on the user's local machine after VM launch:

1. Install system packages: Python, Git, OpenSSH client/server tooling, unzip, jq, build tools, Chromium/browser dependencies.
2. Install FuseKit from the local package artifact or released wheel.
3. Install OpenClaw using the same local-prefix installer path FuseKit already uses.
4. Verify:
   - `python --version`
   - `fusekit --version`
   - `openclaw --version`
   - `openclaw doctor --non-interactive`
   - Playwright Chromium smoke test using the shared runner browser cache.
5. Create `/var/lib/fusekit-runner` for transient worker state.
6. Refuse to continue if verification fails, then either self-heal packages or return to a retryable human gate.

The public launcher should treat this as a prepared environment contract, not as
best-effort shell setup. A demo-ready run must prove the disposable VM has the
expected x86_64 architecture, supported Ubuntu/browser dependency stack,
FuseKit runner helpers, OpenClaw or the approved browser spine, Playwright
Chromium smoke-test readiness, noVNC, and the shared Chrome provider profile
before the first provider account gate is shown. If any of those checks fail,
FuseKit should stop on a runner repair gate or select a verified prepared image
instead of continuing into provider setup with a brittle partial toolchain.
The proof is stored as `.fusekit/runner_readiness.json` with a
`fusekit.runner-profile.v1` contract naming the required OCI visual-browser
profile, 16 GB memory floor, supported Linux image ids, public noVNC/control-room
ports, loopback VNC/OpenClaw gateway ports, OpenClaw plus Playwright Chromium
browser stack, shared provider profile path, observed OS/memory facts, and the
health checks that passed before setup continued.

The app repo should be uploaded as a tarball over SSH, excluding `.git`, `.env`, existing `.fusekit`, caches, dependency folders, and known secret artifacts. Provider credentials must come from the encrypted vault or supervised handoff, not app files.

The Cloud Shell bootstrap must preserve the user's actual launch intent. Provider targets, DNS scope, live URL checks, LLM settings, UI-inference flags, and the selected FuseKit package are forwarded from the launcher into Cloud Shell and again into the disposable VM when the Cloud Shell lane provisions a nested runner. This avoids the bad magic trick where the clean room starts successfully but forgets which services it was supposed to connect.

Cloud Shell auth is treated as a first-class OCI auth lane. When OCI exposes a config with `authentication_type=instance_principal` and a delegation token file, FuseKit loads the SDK signer from that config and uses the signed Cloud Shell session for provisioning. Existing API-key and security-token profiles remain supported.

## Execution Protocol

The local controller and remote VM should communicate over SSH with a one-time session:

1. Local creates an ephemeral RSA SSH key pair for broad compatibility with OCI images.
2. OCI instance metadata receives only the public key.
3. Local waits for SSH reachability with retry.
4. Local uploads:
   - App tarball.
   - FuseKit manifest, if already generated.
   - Encrypted vault bundle, if already present.
   - A short-lived runner instruction file with no raw secrets.
5. The passphrase is provided over SSH stdin only when the remote worker needs to unlock or save the vault. It must not appear in command arguments, shell history, cloud-init logs, audit logs, receipts, or terminal summaries.
6. Remote runs `fusekit launch` in runner mode:
   - Scan.
   - Plan.
   - Wait for human approval.
   - Run provider handoffs through OpenClaw when needed.
   - Apply GitHub, Vercel, DNS, webhooks, and keys.
   - Verify live URL.
   - Write encrypted vault and redacted receipts.
   - Detonate plaintext remote worker state.
7. Local downloads only:
   - Encrypted vault bundle.
   - Redacted audit log.
   - Redacted receipt.
   - Non-secret runner status.

## Detonation

The OCI lane must support two levels:

```text
fusekit runner detonate --runner oci --scope run
fusekit runner detonate --runner oci --scope workspace
```

`run` deletes remote plaintext worker state while preserving the VM for debugging when explicitly requested.

`workspace` is the default after a successful cloud run:

1. Delete remote OpenClaw state after auth-state capture.
2. Shred/remove remote app checkout, temp files, shell histories, logs that FuseKit controls, and SSH keys.
3. Terminate the instance.
4. Delete the boot volume.
5. Delete NSG/security list, subnet, route table, internet gateway, and VCN created by FuseKit.
6. Revoke the OCI API signing key if FuseKit generated/uploaded it.
7. Delete local OCI session token and generated signing keys unless they were explicitly saved into the encrypted vault for reuse.
8. Preserve only encrypted vault and redacted audit/receipt artifacts.

Every workspace detonation writes `.fusekit/workspace_detonation.json` with a
non-secret list of deleted resource categories and redacted failure summaries. A
successful launch is not allowed to finish green if any OCI or remote-worker
cleanup step reports a `failed.*` deletion key; the detonation checkpoint turns
red so the user knows the clean-room VM was not fully destroyed.
The receipt also includes `fusekit.workspace-detonation-resources.v1`, which
classifies whether the remote worker state, OCI compute instance, and
FuseKit-created network resources were covered. Root tenancy/root compartment
scope is recorded as preserved by design when FuseKit did not create a
throwaway compartment; deleting the user's tenancy or root compartment is never
part of default detonation.
An empty cleanup report is treated as incomplete rather than successful: the
receipt must positively name remote-worker cleanup, OCI VM termination, and every
standard FuseKit-created network resource class before the workspace can be
considered detonated.

Every live Run Record also carries `fusekit.durable-state.v1` evidence. That
evidence names the encrypted/redacted sources required to resume after replacing
the worker, and it names the VM/browser/auth/log surfaces that are intentionally
volatile detonation targets.
The same Run Record also carries `fusekit.detonation-scope.v1`: the public-lane
contract for what must be destroyed, what may survive, and why the run can keep
resuming until completion without leaving FuseKit worker state on the user's
machine or in the OCI workspace. The scope must include browser profiles,
provider-auth scratch, passphrase files, uploaded app archives, control-room and
gateway logs, and the disposable compute workspace. The only survivors are the
encrypted vault and redacted proof artifacts such as job state, checkpoints,
gate events, provider strategies, verification, rollback, detonation receipt,
and the Run Record itself.

The worker-replacement drill is part of the public OCI contract, not an
operator-only exercise. At any supervised gate, FuseKit must be able to stop or
lose the disposable VM, provision a new runner with the same
`fusekit.runner-profile.v1` contract, restore only encrypted/redacted survivor
artifacts, reopen the live control room, and continue from the latest gate or
verifier without using host-machine browser state, shell history, plaintext
provider auth, or VM-local scratch files. If that drill cannot be proven, the
run remains in progress and must not show the "ready for recording" state.

Full OCI tenancy/account deletion is not a default detonation target. It should be a separate destructive workflow with explicit user approval.

## CLI Surface

Target commands:

```text
fusekit runner doctor
fusekit runner authorize oci
fusekit runner plan oci
fusekit runner provision oci
fusekit runner exec oci /path/to/app
fusekit runner receipt
fusekit runner detonate --runner oci --scope workspace
```

Launch integration:

```text
fusekit launch /path/to/app \
  --runner auto \
  --oci-account-mode auto \
  --oci-auth-mode auto \
  --oci-shape auto \
  --oci-region auto \
  --fusekit-package git+https://github.com/owner/fusekit.git
```

`--runner local` must keep the current behavior.

## Acceptance Criteria

- A user with no local OCI CLI, no Terraform, no browser automation setup, and no OpenClaw install can run the guided OCI lane after installing FuseKit.
- A user with an existing OCI account can authorize without signing up again.
- A user with an existing OCI config can reuse it when selected or discovered by the runner broker.
- FuseKit can provision a live VM, SSH into it, bootstrap itself and OpenClaw, run the setup worker, retrieve encrypted/redacted artifacts, and destroy the workspace.
- Capacity exhaustion does not crash the run; it becomes a retryable service-side gate with alternatives.
- The encrypted vault opens only with the passphrase.
- Raw OCI keys, SSH keys, provider tokens, LLM auth state, webhook secrets, DNS credentials, and app secrets never appear in app files, logs, receipts, terminal summaries, or detonation survivors.
- `pytest`, `ruff`, and `mypy` pass.
